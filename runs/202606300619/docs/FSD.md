# Business Processing Rules and Functional Specification\n\n## HABADTE (HABADTE)\n\n### For Java / Spring Boot / SQL Server Migration\n\n## 1. Business Purpose\n\nThis system processes patient master, ranking, benefits, institution, and transfer data to produce a consolidated report and XML output. It is used by clinical operations, billing, and administrative staff to understand patient status, benefit eligibility, and institution hierarchy. The processing typically runs in batch or on-demand to prepare reports and downstream feeds.\n\n## 2. Inputs / API Request Parameters\n\nIn the modernized API, inputs correspond to HABADTE entry parameters and job context fields.\n\n| Parameter | AS400 Field | Type | Description |\n|-----------|------------|------|-------------|\n| patientId | PATID | String | Patient identifier used as primary key |\n| institutionId | INSTID | String | Institution key for scoping data |\n| runDate | LDA date field | LocalDate | Processing date from local data area |\n| includeTransfers | Control flag | Boolean | Whether to include transfer history |\n| includeBenefits | Control flag | Boolean | Whether to include benefit data |\n| includeXml | Control flag | Boolean | Whether to produce XML output |\n\nRuntime context mapping to Spring Boot security:\n\n- User identity and roles in Spring Security map to LDA user/job fields and control which PHI files (OMPMAST, HAPTRFR, HXPDICT, OXPBNFIT, OAPIRNK) can be accessed.\n\n## 3. Organizational Hierarchy\n\nA multi-level organization hierarchy is detected via HXPLVL1–HXPLVL6.\n\n| Level | Name | Key Size | Table |\n|-------|------|----------|-------|\n| 1 | Org Level 1 | 4 | HXPLVL1 |\n| 2 | Org Level 2 | 4 | HXPLVL2 |\n| 3 | Org Level 3 | 4 | HXPLVL3 |\n| 4 | Org Level 4 | 4 | HXPLVL4 |\n| 5 | Org Level 5 | 4 | HXPLVL5 |\n| 6 | Org Level 6 | 4 | HXPLVL6 |\n\n## 4. Primary Data Source\n\nPrimary data is drawn from the patient master and ranking tables.\n\n- SQL equivalent table name: dbo.PatientMaster (OMPMAST), dbo.PatientRanking (OAPIRNK).\n- Data access pattern: keyed chain on PATID, sequential scan or ordered read on ranking.\n\nExample SQL Server SELECT with ORDER BY:\n\nSELECT p.PATID, p.NAME, p.DOB, r.RANK, r.EFFDATE\nFROM dbo.PatientMaster p\nLEFT JOIN dbo.PatientRanking r ON r.PATID = p.PATID\nWHERE p.PATID = @patientId\nORDER BY r.RANK ASC;\n\nPrimary file key fields:\n\n| File | Key Fields |\n|------|-----------|\n| OMPMAST | PATID, MRN |\n| OAPIRNK | PATID, RANK |\n\n## 5. Patient/Record Inclusion and Exclusion Rules\n\n### Rule BR-017\n\n- BR-ID: BR-017\n- Description: When -FILE INDICATOR equals zero, branch to "SKIP".\n- Pseudocode IF condition:\n\nIF fileIndicator = 0 THEN\n  SKIP RECORD;\nEND IF;\n\n- SQL WHERE clause equivalent:\n\nWHERE fileIndicator <> 0\n\n### Rule BR-018\n\n- BR-ID: BR-018\n- Description: When -FLAG INDICATOR equals void/voided, branch to "SKIP".\n- Pseudocode IF condition:\n\nIF flagIndicator = 'V' THEN\n  SKIP RECORD;\nEND IF;\n\n- SQL WHERE clause equivalent:\n\nWHERE flagIndicator <> 'V'\n\n### Rule BR-019\n\n- BR-ID: BR-019\n- Description: When -INPATIENT/OUTPATIENT FLAG equals outpatient, branch to "SKIP".\n- Pseudocode IF condition:\n\nIF inOutFlag = 'O' THEN\n  SKIP RECORD;\nEND IF;\n\n- SQL WHERE clause equivalent:\n\nWHERE inOutFlag <> 'O'\n\n### Combined Summary SQL WHERE Clause\n\nWHERE fileIndicator <> 0\n  AND flagIndicator <> 'V'\n  AND inOutFlag <> 'O'\n\n-- BR-017: fileIndicator <> 0\n-- BR-018: flagIndicator <> 'V' (not voided)\n-- BR-019: inOutFlag <> 'O' (exclude outpatient)\n\n## 6. Data Enrichment Steps\n\nEach record enriched with secondary lookups.\n\n### Enrichment 1 — Institution Details\n\n- BR-ID: (implicit)\n- Description: Lookup institution name and org hierarchy.\n- Key Fields: INSTID.\n- SELECT query equivalent:\n\nSELECT i.NAME, i.LVL1KEY, i.LVL2KEY\nFROM dbo.Institution i\nWHERE i.INSTID = @instId;\n\n- Delete-flag and effective-date logic: Only active institutions (STATUS = 'A').\n- Derived flags: orgLevel1Name, orgLevel2Name.\n\n### Enrichment 2 — Benefits\n\n- BR-ID: (implicit)\n- Description: Lookup patient benefits.\n- Key Fields: PATID.\n- SELECT query equivalent:\n\nSELECT b.BNFTID, b.STATUS, b.INSTID\nFROM dbo.Benefit b\nWHERE b.PATID = @patientId AND b.STATUS = 'A';\n\n### Enrichment 3 — Dictionary Values\n\n- BR-ID: (implicit)\n- Description: Lookup patient dictionary codes.\n- Key Fields: PATID.\n- SELECT query equivalent:\n\nSELECT d.CODE, d.VALUE\nFROM dbo.PatientDictionary d\nWHERE d.PATID = @patientId;\n\n## 7. Counting and Aggregation Rules\n\nCounters derived from calculation-type rules and utility programs.\n\n| Counter | Incremented When | Description |\n|---------|------------------|-------------|\n| totalPatients | Each included patient record | Count of patients processed |\n| totalBenefits | Each active benefit found | Count of active benefits per patient |\n| totalTransfers | Each transfer record read | Count of transfers per patient |\n| totalXmlRecords | Each XML row written | Count of XML payload records |\n\n## 8. Output Data Structure\n\n### 8.1 Header / Response Header Fields\n\n- runDate\n- institutionId\n- institutionName\n\n### 8.2 Detail Row Fields\n\n| Field | Source | Type | Description |\n|-------|--------|------|-------------|\n| patientId | OMPMAST.PATID | String | Patient identifier |\n| patientName | OMPMAST.NAME | String | Patient name |\n| dob | OMPMAST.DOB | LocalDate | Date of birth |\n| rank | OAPIRNK.RANK | Integer | Ranking value |\n| benefitCount | Benefit rows | Integer | Number of active benefits |\n| transferCount | HAPTRFR rows | Integer | Number of transfers |\n\n### 8.3 Summary/Footer Fields\n\n- totalPatients\n- totalBenefits\n- totalTransfers\n- totalXmlRecords\n\n### 8.4 Sort Order\n\n- Primary sort: rank ascending.\n- Secondary sort: patientId ascending.\n\n## 9. Complete Processing Flow Step-by-Step\n\n1. Initialization\n   - Load LDA (job context) via HXXLDA.\n   - Initialize control structures via HXXCNTRL.\n2. Load Preferences\n   - Resolve application preferences via HXXAPPPRFP/HXXAPPPRF and HXPTABLD/HXLTABLS.\n3. Resolve Context Data\n   - Determine institution and org hierarchy via HXPLVL1–HXPLVL6 and OXPNSTN/HXPNSTN.\n4. Query Primary Data\n   - Read patient master from OMPMAST/HMLMAST5H.\n   - Read ranking from OAPIRNK/HAPIRNK.\n5. Per-Record Enrichment\n   - Validate dates using XFXCYMD and XFXLEAP.\n   - Apply counters via XFXCNTR.\n   - Lookup benefits via OXPBNFIT/HXPBNFIT.\n   - Lookup dictionary values via HXPDICT.\n   - Apply filter rules BR-017–BR-019 to include or skip records.\n6. Assemble Response\n   - Build XML payload using CXXXMLP/HXXXML and write to HXPXMLR.\n   - Print report via PRINTER.\n   - Return aggregated response to caller.\n\n## 10. Data Type Conversions\n\n### 10.1 Date Fields\n\n- DECIMAL(8,0) YYYYMMDD -> Java LocalDate via parsing year, month, day.\n\n### 10.2 Time Fields\n\n- HHMM integer or string -> Java LocalTime via hour/minute extraction.\n\n### 10.3 Packed Decimal Keys\n\n- Packed decimals used for keys (e.g., RANK) -> Long or BIGINT in SQL Server.\n\n### 10.4 String Trimming\n\n- CHAR fields -> Java String with right-trim of spaces.\n\n## 11. SQL Server Table Mapping\n\n| AS400 Object | SQL Server Table | Used For |\n|-------------|------------------|---------|\n| OMPMAST | dbo.PatientMaster | Core patient data |\n| OAPIRNK | dbo.PatientRanking | Ranking data |\n| OXPBNFIT | dbo.Benefit | Benefit data |\n| OXPNSTN | dbo.Institution | Institution data |\n| HAPTRFR | dbo.PatientTransfer | Transfer data |\n| HXPDICT | dbo.PatientDictionary | Dictionary data |\n| HXPTABLD | dbo.AppTableDef | Preferences and status codes |\n| HXPLVL1–HXPLVL6 | dbo.OrgLevel1–6 | Org hierarchy |\n\nSuggested covering indexes:\n\n- dbo.PatientRanking: IX_PatientRanking_PatientRank (PATID, RANK).\n- dbo.Benefit: IX_Benefit_PatientStatus (PATID, STATUS).\n- dbo.PatientTransfer: IX_PatientTransfer_PatientDate (PATID, TRFDATE).\n- dbo.PatientDictionary: IX_PatientDictionary_PatientCode (PATID, CODE).\n\n## 12. Spring Boot API Design\n\nRecommended REST endpoint:\n\n- GET /api/patient-report\n\nQuery parameters:\n\n- patientId (required)\n- institutionId (optional)\n- includeTransfers (optional)\n- includeBenefits (optional)\n- includeXml (optional)\n\nLayer structure:\n\n- Controller: PatientReportController\n- Service: PatientReportService\n- Repository: PatientRepository, RankingRepository, BenefitRepository, InstitutionRepository, TransferRepository, DictionaryRepository, PreferenceRepository\n\nResponse JSON shape sample:\n\n{\n  "runDate": "2026-06-30",\n  "institutionId": "INST01",\n  "institutionName": "Main Hospital",\n  "patients": [\n    {\n      "patientId": "P123",\n      "patientName": "Jane Doe",\n      "dob": "1980-01-01",\n      "rank": 1,\n      "benefitCount": 2,\n      "transferCount": 0\n    }\n  ],\n  "summary": {\n    "totalPatients": 1,\n    "totalBenefits": 2,\n    "totalTransfers": 0,\n    "totalXmlRecords": 1\n  }\n}\n\nEntity/DTO sketches:\n\n- Request DTO: PatientReportRequest (patientId, institutionId, flags).\n- Detail DTO: PatientReportDetail (patientId, name, dob, rank, counts).\n- Response DTO: PatientReportResponse (header, list of details, summary).\n\n## 13. Performance Considerations\n\nTo avoid N+1 queries, replace per-record loops with JOINs.\n\nExample recommended query:\n\nSELECT p.PATID, p.NAME, p.DOB, r.RANK, b.BNFTID, t.TRFDATA\nFROM dbo.PatientMaster p\nLEFT JOIN dbo.PatientRanking r ON r.PATID = p.PATID\nLEFT JOIN dbo.Benefit b ON b.PATID = p.PATID AND b.STATUS = 'A'\nLEFT JOIN dbo.PatientTransfer t ON t.PATID = p.PATID\nWHERE p.PATID = @patientId\n  AND fileIndicator <> 0\n  AND flagIndicator <> 'V'\n  AND inOutFlag <> 'O';\n\n## 14. Business Rules Reference Summary\n\n| Rule ID | Description |\n|---------|-------------|\n| BR-017 | When -FILE INDICATOR equals zero, branch to 'SKIP' |\n| BR-018 | When -FLAG INDICATOR equals void/voided, branch to 'SKIP' |\n| BR-019 | When -INPATIENT/OUTPATIENT FLAG equals outpatient, branch to 'SKIP' |\n\n## 15. Edge Cases\n\n| Scenario | Expected Behavior |\n|----------|------------------|\n| fileIndicator = 0 | Record is skipped (BR-017) |\n| flagIndicator = 'V' | Record is skipped (BR-018) |\n| inOutFlag = 'O' | Record is skipped (BR-019) |\n| No ranking record found | Rank defaults to null or 0; record still processed |\n| No benefits found | benefitCount = 0 |\n| No transfers found | transferCount = 0 |\n| External text files missing | System uses PF data only; logs warning about missing feeds |
