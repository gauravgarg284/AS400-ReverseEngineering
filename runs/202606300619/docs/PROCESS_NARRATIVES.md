# Process Narratives — HABADTE\n\n## DATA_MAINTENANCE Domain\n\n### Overview\n\nThe DATA_MAINTENANCE domain encompasses utility programs responsible for validating dates, managing counters, performing table lookups, and loading descriptions. These utilities are reused across workflows to ensure consistent data quality and control.\n\n### Trigger\n\nProcesses in this domain are triggered whenever HABADTE or other business programs need to validate a date, manage a counter, or look up configuration values.\n\n### Process Flow\n\n1. A calling program (such as HABADTE) prepares input parameters (e.g., date components, counters, table IDs).\n2. The appropriate utility program is called (XFXCYMD for dates, XFXCNTR for counters, XFXLDSC/XFXTABL for table lookups).\n3. The utility applies its business rules (BR-001–BR-016).\n4. The utility returns status indicators and possibly transformed values to the caller.\n5. The caller uses the returned status to decide whether to continue processing or exit.\n\n### Business Rules Applied\n\n- BR-001–BR-002: Counter exit rules in XFXCNTR.\n- BR-003–BR-008: Date validation rules in XFXCYMD.\n- BR-009–BR-012: LDAMAP validation rules in XFXLDSC.\n- BR-013–BR-016: Table indicator exit rules in XFXTABL.\n\n### Error Handling\n\n- Invalid dates (e.g., year out of range, month/day invalid) cause the utilities to branch to EXIT, returning an error indicator to the caller.\n- Counter values outside allowed ranges cause early exit from loops.\n- Table lookups that fail or hit indicator conditions cause early exit, signaling configuration issues.\n\n### Data Used\n\n- Internal parameters passed by callers (dates, counters, table IDs).\n- Configuration tables via HXPTABLD and related logical files.\n\n### Outputs Produced\n\n- Status indicators (valid/invalid, exit flags).\n- Possibly transformed values (e.g., normalized dates).\n\n## PATIENT_MANAGEMENT Domain\n\n### Overview\n\nThe PATIENT_MANAGEMENT domain covers HABADTE and related programs that orchestrate patient master, ranking, benefits, institution, and transfer data to produce reports and XML outputs. It is central to patient-centric reporting and operational decision-making.\n\n### Trigger\n\nProcessing is triggered by a user request or scheduled job to generate a patient report or batch of reports for a given institution or patient set.\n\n### Process Flow\n\n1. Initialization: Job context and control structures are initialized via LDA and control copy members.\n2. Preference Load: Application preferences are resolved using HXXAPPPRF and table PF/LF objects.\n3. Context Resolution: Institution and org hierarchy are determined via OXPNSTN/HXPNSTN and HXPLVL1–HXPLVL6.\n4. Primary Data Query: Patient master and ranking data are retrieved from OMPMAST/HMLMAST5H and OAPIRNK/HAPIRNK.\n5. Per-Record Enrichment: Benefits, dictionary entries, and transfers are loaded; date and counter utilities are invoked.\n6. Filtering: BR-017–BR-019 are applied to exclude records that should not be processed or reported.\n7. Output Assembly: XML payloads are built and written to HXPXMLR, and printed reports are generated via PRINTER.\n\n### Business Rules Applied\n\n- BR-017: File indicator skip.\n- BR-018: Void flag skip.\n- BR-019: Outpatient skip.\n\n### Error Handling\n\n- Records failing filter rules are skipped rather than processed.\n- Missing external text files (TAPIRNK, TMPMAST, TXPBNFIT, TXPNSTN) result in reliance on PF data and logging of warnings.\n- Utility errors from DATA_MAINTENANCE domain cause early exit or record-level skips.\n\n### Data Used\n\n- Patient master (OMPMAST, HMLMAST5H).\n- Patient ranking (OAPIRNK, HAPIRNK).\n- Benefits (OXPBNFIT, HXPBNFIT).\n- Institutions (OXPNSTN, HXPNSTN).\n- Transfers (HAPTRFR).\n- Dictionary entries (HXPDICT).\n- Org hierarchy (HXPLVL1–HXPLVL6).\n- Preferences and status codes (HXPTABLD, HXLTABLD, HXLTABLP, HXLTABLS).\n\n### Outputs Produced\n\n- Printed patient reports via PRINTER.\n- XML payloads in HXPXMLR for downstream systems.\n- Aggregated counts and summary metrics.\n
