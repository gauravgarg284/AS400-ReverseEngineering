# Low-Level Design (LLD) — HABADTE\n\n## Section 1 — Architecture Overview\n\n### 1.1 Technology Stack\n\n| Layer | Technology | Description |\n|------|-----------|-------------|\n| Application | RPGLE / SQLRPGLE | Core business logic (HABADTE main, utility programs, preference driver) |\n| Data Definition | DDS PF/LF | Physical and logical files for patients, institutions, benefits, tables, org levels, XML |\n| Job Context | Data Areas / LDA | Local data area defined via HXXLDA copy member |\n| Configuration | Table PF/LF | HXPTABLD and HXLTABL* logical files for status codes and preferences |\n| External Interfaces | Printer files, text exports | TAPIRNK.txt, TMPMAST.txt, TXPBNFIT.txt, TXPNSTN.txt, printer/PRINTER file |\n| Security | IBM i object-level security | PHI files protected via library/object authorities |\n\n### 1.2 Component Interaction\n\nHABADTE acts as the central orchestrator. It:\n\n- Reads patient master data from OMPMAST and logical views such as HMLMAST5H.\n- Reads patient ranking from OAPIRNK and HAPIRNK.\n- Reads benefits from OXPBNFIT and HXPBNFIT.\n- Reads institution data from OXPNSTN and HXPNSTN.\n- Reads patient dictionary/demographics from HXPDICT.\n- Reads patient transfer history from HAPTRFR.\n- Uses HXPTABLD and HXLTABL* logical files to resolve status codes, preferences, and table-driven behavior.\n- Uses HXPLVL1–HXPLVL6 to resolve organizational hierarchy and preference overrides.\n- Delegates reusable logic to utility programs (XFXCNTR, XFXCYMD, XFXLDSC, XFXTABL, XFXGETID, XFXLEAP, XFXMRNROL).\n- Delegates preference resolution to HXXAPPPRF (directly or via copy members).\n- Produces output via printer/PRINTER and XML PF HXPXMLR, using XML copy members (CXXXMLP, HXXXML).\n\n### 1.3 Design Patterns in Use\n\n- Call Patterns: HABADTE uses external calls to utility programs for date validation, counter logic, table lookups, and MRN rollover, following a procedural service pattern.\n- Data Coupling: Programs are tightly coupled to PF/LF structures via F-specs; logical files provide filtered/ordered views but share formats with PFs.\n- Error Handling: Indicator-based branching (e.g., *IN79, file indicators, flag indicators) is used to short-circuit processing (see BR-013–BR-019).\n- Initialization: HABADTE initializes control structures via HXXCNTRL and job context via HXXLDA, then loads preferences via HXXAPPPRFP/HXXLEVEL.\n- Framework Usage: Table-driven configuration via HXPTABLD and its logical files acts as a lightweight framework for status/parameter management.\n\n## Section 2 — Database Schema\n\n### 2.1 Physical Files (PF)\n\n#### 2.1.1 OMPMAST — Patient Master\n\n| Column | DB2 Type | Length | Dec | Logical Files (LF)\n\n#### 2.2.1 HMLMAST5H — Patient Master Logical\n\n- Based on PF: OMPMAST\n- Key Fields: PATID, MRN\n- Select/Omit: May select active patients or a specific subset (e.g., hospital 5H).\n\n#### 2.2.2 HAPIRNK — Patient Ranking Logical\n\n- Based on PF: OAPIRNK or TAPIRNK equivalent.\n- Key Fields: PATID, RANK.\n- Select/Omit: May omit voided or inactive ranking records.\n\n#### 2.2.3 HXPBNFIT — Benefit Logical\n\n- Based on PF: OXPBNFIT.\n- Key Fields: PATID, BNFTID.\n- Select/Omit: May select only active benefits.\n\n#### 2.2.4 HXPNSTN — Institution Logical\n\n- Based on PF: OXPNSTN.\n- Key Fields: INSTID.\n- Select/Omit: May select only active institutions.\n\n#### 2.2.5 HXLTABLD, HXLTABLP, HXLTABLS — Table Logical Files\n\n- Based on PF: HXPTABLD.\n- HXLTABLD: General table view keyed by TABLEID and KEY.\n- HXLTABLP: Parameterized view keyed by TABLEID and PARAM.\n- HXLTABLS: Status view keyed by STATUS.\n\n## Section 3 — Status and Type Reference Data\n\nStatus codes appear in CABEQ-style literals and table-driven logic via HXLTABLS. Examples include:\n\n| Status Code | Meaning | Source |\n|-------------|---------|--------|\n| A | Active | HXLTABLS / CABEQ literals |\n| I | Inactive | HXLTABLS / CABEQ literals |\n| V | Voided | HABADTE flag indicator (BR-018) |\n| O | Outpatient | HABADTE in/outpatient flag (BR-019) |\n\nThese codes are resolved via table lookups (XFXTABL, HXPTABLD/HXLTABLS) and control branching in HABADTE.\n\n## Section 4 — Stored Procedure Logic Mappings\n\nTop subroutines with fan_in greater than 1 (conceptual mapping for migration):\n\n| Program | Subroutine | Parameters | Files Affected |
